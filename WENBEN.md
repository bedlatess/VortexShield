# VortexShield (涡流之盾) 项目白皮书与后续演进路线

## 1. 项目定位

VortexShield 是一套面向登录、注册、支付、内容发布、接口访问等高风险场景的智能行为验证码与人机风控系统。它的目标不是简单地“让用户做一道题”，而是构建一条从客户端环境识别、行为轨迹采集、加密传输、服务端风控、安全校验编排、数据沉淀到 AI 模型训练的完整安全闭环。

在产品体验上，VortexShield 采用类似 Cloudflare Turnstile 的无感优先策略：低风险用户直接静默通过；中风险用户展示轻量复选框；高风险用户才进入滑块拼合校验。这样的设计可以在安全性和用户体验之间取得更好的平衡，避免所有用户都被迫面对高摩擦验证码。

## 2. 核心能力概览

### 2.1 多态安全校验编排

系统内置三种验证形态：

- `SILENT`：无感静默验证，适用于干净浏览器环境和低风险会话。
- `CLICK_CHECKBOX`：类 Turnstile 复选框验证，适用于信号不完整但未发现明确攻击特征的用户。
- `SLIDER`：形状匹配滑块验证，适用于命中 webdriver、自动化脚本、异常指纹等高风险环境。

后端通过 `/api/captcha/precheck` 对前端提交的环境探针进行初筛，并返回明确的 `captcha_type`，前端 SDK 根据类型自动渲染对应 UI。

### 2.2 混合密码学通信链路

VortexShield 使用 `RSA-OAEP-2048 + AES-GCM-256` 混合加密机制保护验证 Payload：

- 后端生成短时 RSA 密钥对，只下发公钥。
- 前端浏览器本地随机生成 AES-256 会话密钥。
- 行为轨迹、点击/滑块坐标、环境指纹等敏感数据使用 AES-GCM 加密。
- AES 会话密钥使用 RSA-OAEP 公钥封装。
- 后端先用私钥解开 AES 密钥，再解密 Payload。

这条链路可以降低中间人篡改、明文密钥泄露和接口重放的风险。

### 2.3 生物动力学风控引擎

系统对前端采集的轨迹点 `[x, y, timestamp, event_type]` 进行统计分析，提取以下特征：

- 轨迹总耗时。
- X/Y 轴位移。
- 一阶速度 `v`。
- 二阶加速度 `a`。
- 速度标准差。
- 最大瞬时速度。
- 加速度方差。
- 时间间隔波动。

这些特征可以识别常见机器行为，例如匀速直线拖动、瞬间坐标跳跃、极短事件序列和自动化环境探针命中。

### 2.4 动态滑块容错

滑块验证不再使用粗暴的固定像素误差，而是使用面积重合率：

```text
overlap_ratio = max(0, 1 - abs(slider_x - target_x) / piece_width)
```

最终判定结合轨迹风险分值：

- 高风险轨迹直接拒绝。
- 低风险真人轨迹放宽到 70% 重合即可通过。
- 普通模糊轨迹要求 85% 重合。

这可以明显降低真人误杀，同时保留对机器代滑的拦截能力。

### 2.5 数据飞轮与 AI 训练管道

验证完成后，系统会把轨迹样本写入 `logs/trajectory_data.jsonl`，供后续训练使用。当前已经具备：

- 指纹脱敏。
- 轨迹样本异步写入。
- 防投毒过滤器。
- 10MB 日志轮转。
- 最多保留 5 个备份文件。
- 真人正确对齐与真人未对齐的引导样本生成脚本。
- 基于 `RandomForestClassifier` 的训练脚本。
- 模型输出到 `models/vsec_rf_model.pkl`。

引导样本通过 `scripts/seed_training_logs.py` 写入，并使用 `bootstrap_human_passed`、`bootstrap_human_slider_miss` 等原因字段与真实生产样本区分。未来可以逐步让机器学习模型接管部分风险判定逻辑，从硬编码规则演进为规则与模型结合的风控体系。

## 3. 当前系统架构

```text
Browser / SDK
  ├─ 环境探针采集
  ├─ Canvas / WebGL / webdriver 检测
  ├─ 鼠标与拖拽轨迹采集
  ├─ RSA + AES 混合加密
  └─ 多态 UI 渲染

FastAPI Backend
  ├─ /precheck-key  下发预检公钥
  ├─ /precheck      静默风险分流
  ├─ /challenge     显式滑块校验流程
  └─ /verify        解密、校验、风控、签发凭证

Ops Console
  ├─ /home/api          接入凭证中心
  ├─ /home/admin/login  项目管理员登录
  ├─ /home/admin        科技风安全运营面板
  └─ /home/admin/api    站点启停、删除、secret 轮换与运行态查看

Risk Engine
  ├─ 环境风险评估
  ├─ 轨迹统计特征提取
  ├─ 硬规则拦截
  └─ 动态滑块容错决策

Data Flywheel
  ├─ JSONL 样本沉淀
  ├─ 防投毒过滤
  ├─ 日志轮转
  └─ RandomForest 模型训练
```

## 4. 典型业务场景

VortexShield 可以用于以下场景：

- 用户登录前的人机验证。
- 注册和短信验证码发送前的防刷验证。
- 支付、提现、改密等敏感操作二次校验。
- 评论、发帖、投票等内容入口防自动化灌水。
- 开放 API 的轻量行为风控前置层。
- 内部系统防自动化撞库与弱口令扫描。

## 5. 安全边界说明

VortexShield 当前版本已经具备较完整的验证码与行为风控链路，但在真实生产环境中仍建议注意以下边界：

- 内存 Session Store 适合单机演示，生产多副本部署应迁移到 Redis。
- 当前 Docker 默认单 worker，保证内存 RSA 私钥与验证码会话在 precheck -> verify 链路中稳定命中同一进程。
- 控制台签发的站点 API 已支持本地 JSON 持久化，默认写入 `data/site_registry.json`，避免服务重启后接入凭证丢失。
- `/home/admin/login` 是 VortexShield 项目管理员入口，使用 HttpOnly Cookie 保存管理员会话，不面向普通接入方开放。
- IP 限流为单进程内存版，多实例部署应迁移到 Nginx、API Gateway 或 Redis 令牌桶。
- 机器学习样本需要长期真实数据积累，早期模型不可完全替代硬规则。
- 指纹采集需要遵守所在地区隐私合规要求，例如最小化采集、脱敏存储和保留周期控制。
- RSA 私钥、签名密钥和日志数据应纳入生产密钥管理和审计体系。

## 6. 后续演进路线

### 6.1 存储与集群化

- 将 `session_store.py` 从内存 Mock Store 切换为 Redis。
- 将 verify signature 存储迁移到 Redis，并支持 TTL 自动过期。
- 将 `site_registry.py` 从本地 JSON 文件升级为数据库或配置中心，支持审计、禁用、轮换和多租户管理。
- 在 Docker Compose 中启用 Redis profile，形成可水平扩展架构。
- 支持多实例部署下的统一限流与统一校验状态。

### 6.2 风控模型增强

- 将 `train_risk_model.py` 的特征提取逻辑抽成公共模块，供离线训练和在线推理复用。
- 在 `risk_engine.py` 中加载 `models/vsec_rf_model.pkl`，加入 `predict_proba` 风险概率。
- 使用规则优先、模型复核的双层决策：
  - 硬规则命中直接拦截。
  - 模型高风险进入滑块校验流程或拒绝。
  - 模型低风险降低用户交互强度。
- 引入更多模型，例如 XGBoost、LightGBM、Isolation Forest 或序列模型。

### 6.3 数据治理与反投毒

- 给 JSONL 样本增加数据来源、业务场景、版本号和决策路径。
- 增加样本去重，避免同一攻击脚本批量污染训练集。
- 建立人工审核集和灰度样本池。
- 对训练样本做分层采样，避免某一类攻击流量压倒真实用户样本。
- 增加日志上传对象存储或 Kafka 流式处理能力。

### 6.4 前端 SDK 产品化

- 将 SDK 打包为 npm 包。
- 增加主题配置、语言包扩展和可插拔 UI。
- 支持移动端触摸轨迹优化。
- 增加无障碍能力，例如键盘操作和屏幕阅读器提示。
- 支持业务方自定义回调、埋点和错误码处理。

### 6.5 安全加固

- 引入服务端签名密钥轮换。
- 增加 HMAC 或 JWT 格式的 `verify_signature`。
- 增加校验 token 与业务 action、用户 ID、IP、UA 的绑定。
- 配合 Nginx / Cloudflare / API Gateway 做边缘层限流。
- 对前端 SDK 增加完整性校验和版本签名。

### 6.6 运维与可观测性

- 增加 Prometheus 指标：
  - precheck 通过率。
  - 增强校验升级率。
  - verify 成功率。
  - bot 拦截率。
  - 平均验证耗时。
  - 日志丢弃数量。
- 增加 Grafana Dashboard。
- 增加结构化业务审计日志。
- 增加管理员面板操作审计，例如启停、删除、secret 轮换和登录失败事件。
- 增加 Docker healthcheck。
- 增加 CI/CD 自动测试和镜像构建流程。

## 7. 工程与安全价值

VortexShield 的核心价值在于把验证码从“单点交互组件”升级为“可持续进化的行为安全平台”：

- 对普通用户低打扰，提升转化率。
- 对自动化脚本高强度拦截，降低撞库、薅羊毛和灌水风险。
- 对安全团队提供可训练数据，形成长期数据壁垒。
- 对业务系统提供统一验证凭证，便于接入登录、注册、支付等多个场景。

如果后续继续完善 Redis 集群化、模型在线推理、可观测性和 SDK 产品化，VortexShield 可以从技术作品集进一步演进为可私有化部署的企业级人机验证服务。
